DEVELOPMENT TIMELINE
How the work unfolded — and how it was measured

This project started as an experiment: how far could generative-AI tooling get on a real, correctness-sensitive task — turning a NASFAA flowchart PDF into a verified decision engine? Measuring the shape of that effort — what was built, when, how much human-active time it took, and how much carried an AI co-author — is in large part the point of the whole thing. This page is that record, and it is deliberately explicit about how every number is defined and what is still uncertain.

Every figure is derived from git log by bin/timeline (phase table) and bin/time-analysis (the repo-wide hours oracle). The table is a snapshot; regenerate it with make timeline after new commits land.

Definitions

Commit
The atomic unit of measurement. Every figure here is derived from git log; one commit is one row of raw data.
Phase
A thematic grouping of commits by which aspect of the work they advance (initial implementation, rule engine, CLI, web, tooling, …). Thematic, not chronological — phases overlap in calendar time.
Classification
Each commit is assigned to exactly one phase by an ordered, first-match ruleset over its subject line, plus an override table for documented judgment calls. A commit matching no rule is reported as unclassified and never force-fit. Currently: 0 unclassified.
Active hours
Within-session wall-clock time: commits are grouped into sessions, each session's duration is (last commit − first commit), and active hours is the sum. A lower bound on real effort — time that doesn't end in a commit is invisible.
Session (60-minute gap)
A run of commits where each is ≤ 60 minutes after the previous. A gap > 60 minutes starts a new session. A commit with no neighbour within 60 minutes is a singleton session contributing ≈ 0 h.
Churn (+/−)
Lines added / removed, counting source code and correctness-bearing YAML, and excluding generated bundles (web/*/data.js, *.json), docs (*.md, *.pdf), and lockfiles.
AI %
The fraction of a phase's commits carrying a Claude/AI Co-Authored-By trailer. It is the only machine-readable signal of AI involvement — it sees trailers, nothing else.

Decisions

Thematic phases over chronological buckets
The work was not linear: correctness fixes, docs, and tooling run alongside every feature phase. Calendar buckets would scatter one conceptual effort across many; thematic phases keep "the rule engine" or "the web pages" as single readable units — at the cost of overlapping date ranges.
Active hours reported as a floor
A commit marks when work was saved, not when it started. Reading the PDF, planning routing, and debugging that doesn't end in a commit are uncounted. Rather than guess a multiplier, every hours figure is labelled a lower bound.
60-minute session gap
Long enough to bridge normal think/edit pauses, short enough that an overnight or multi-day gap isn't counted as continuous work. It is a knob, not a law — a different gap changes every hours figure.
Whole-repo total ≠ sum of phase hours
Sessionizing within a phase drops the gap between two consecutive commits that belong to different phases. So the per-phase sum (19.90 h) is below the whole-repo sessionization (24.95 h). The Totals row, computed across all commits, is the repo-wide floor.
Churn excludes generated/derived files
Generated bundles, docs, and lockfiles inflate line counts without representing authored logic. Correctness-bearing YAML (rules, questions, scenarios) is counted as code, because editing it is a correctness change, not data entry.
AI % from co-author trailers
The only objective, after-the-fact signal available. Known blind spot: the pre-gem Cursor era (Phase 0) predates the trailer convention, so it reads 0% despite being heavily AI-assisted. The metric measures attributed authorship, not actual assistance.
Override table for judgment calls
A few commits whose subject lines would misroute under the ruleset are pinned to the correct phase by hash, documented in bin/timeline. This keeps the classification deterministic and auditable rather than hand-massaged.

Open questions

These are unresolved — listed so the numbers aren't read as more precise than they are.

  1. What multiplier turns "active hours" (a floor) into real elapsed effort? Reading, planning, and dead-end debugging are uncounted, so the true figure is unknown — only bounded below.
  2. The "Refactor / quality" phase reads 0.00 h because its two commits are > 60 min apart (two singleton sessions). Isolated or widely-spaced commits are systematically undercounted — how should they be valued?
  3. AI % undercounts early AI use (Phase 0 = 0% but was Cursor-driven). Can AI involvement before the trailer convention be reconstructed at all?
  4. Phase assignment is per-commit and thematic; with overlapping phases, some commits could defensibly belong to two. How much does the boundary choice move the numbers?
  5. The 60-minute session gap is arbitrary. What gap best matches actual working sessions, and how sensitive are the totals to it?
  6. A commit is a coarse proxy for effort — a one-line fix and a 500-line feature each count as one. Should effort be weighted (by churn? by review time?) rather than counted?
  7. None of this captures AI cost: tokens, retries, or discarded output. A full accounting of AI-assisted development would include the work that never landed.
