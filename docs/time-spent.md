## Method

Sessions are inferred by grouping consecutive commits where the gap
between commits is ≤ 60 minutes; session duration is `last_commit_time
− first_commit_time`.  Gaps larger than 60 minutes start a new session.
Single-commit "sessions" report a 0:00:00 duration — they are commits
the heuristic can't time, not commits with no work behind them.

Churn excludes generated build artifacts (`web/*/data.js`,
`web/*/*.json` rules/questions/scenarios bundles, `web/text-verify/
data.js`), documentation (`*.md`, `*.pdf`), and lockfiles
(`Gemfile.lock`).  YAML data files (`nasfaa_rules.yml`,
`nasfaa_questions.yml`, `nasfaa_scenarios.yml`) **are** counted because
an incorrect rule there has the same impact as a code bug.

## Approaches for further investigation

The commit-gap heuristic systematically undercounts work.  A reading
session, a planning conversation, a long debug session that ends in
a single commit — all collapse to a 0:00:00 row.  Candidate signals
that would tighten the estimate, roughly in order of bang for buck:

1. **Claude Code transcript timestamps.**  Per-message timestamps in
   `~/.claude/projects/-Users-daviddoolin-src-nasfaa/*.jsonl` bound
   each conversation precisely (first user message → last assistant
   reply).  This is the highest-resolution signal available and
   captures all the planning/reading/debugging time that doesn't end
   in a commit.  Caveat: only covers Claude Code sessions; pre-AI work
   and non-Claude editor sessions aren't visible to it.

2. **Git reflog timestamps.**  `git reflog --date=iso` records every
   HEAD movement: commits, amends, checkouts, resets, even commits
   that got rebased away.  Bounds activity more tightly than `git log`
   alone, especially during rebasing and amend-heavy sessions.

3. **Editor telemetry (WakaTime, CodeTime, RescueTime).**  If
   installed, these produce ground-truth keystroke-level activity
   timelines.  Best signal of all but requires having had the tool
   running historically — can't backfill.

4. **File mtime stratification between commits.**  For each commit
   pair (n, n+1), the mtimes of files touched in commit n+1 indicate
   when the in-progress work happened.  Limited by filesystem-level
   mtime preservation across pulls/checkouts.

5. **Pull-request and issue activity.**  Comments, review responses,
   CI re-runs all have timestamps that bracket "thinking about this
   work" time.  Not yet applicable here (no PR/issue history) but
   worth wiring into any CI work that lands next.

6. **Self-instrumentation (Pomodoro / Toggl).**  Manual but accurate;
   only useful going forward.

Of these, **(1)** is the only one that can produce a richer picture
for the work already done — the transcripts already exist.  A future
pass could correlate Claude Code session boundaries with commit
clusters and report both numbers side by side.
